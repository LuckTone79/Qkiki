# 프로젝트 대화 개별 제거 설계

## 목표

프로젝트 상세 화면의 각 대화 카드에서 해당 대화를 개별적으로 정리할 수 있게 한다. 기본 선택은 프로젝트에서만 연결을 해제하는 안전한 동작이며, 사용자가 명시적으로 선택한 경우에만 원본 대화와 결과를 영구 삭제한다.

## 사용자 경험

각 대화 카드의 `열기` 버튼 옆에 `삭제` 버튼을 표시한다. 삭제 버튼을 누르면 다음 세 가지 선택지가 있는 모달을 연다.

1. `프로젝트에서만 제거`: 프로젝트와의 연결 및 같은 프로젝트에 수집된 해당 대화 항목만 제거한다. 원본 대화, 결과, 첨부파일은 보존한다.
2. `원본까지 영구 삭제`: 영구 삭제 경고와 재확인을 한 번 더 표시한 뒤 대화와 종속 데이터를 삭제한다.
3. `취소`: 아무 변경 없이 모달을 닫는다.

모달은 키보드로 조작할 수 있고, 제목과 설명으로 대상 대화를 명확히 식별한다. 처리 중에는 해당 선택 버튼을 비활성화하고 중복 요청을 막는다.

## API와 데이터 처리

프로젝트 연결 해제용 동적 Route Handler를 추가한다.

- 경로: `DELETE /api/projects/[id]/sessions/[sessionId]`
- 인증: 현재 로그인 사용자를 확인한다.
- 소유권: 프로젝트와 대화가 모두 현재 사용자 소유이고, 대화가 해당 프로젝트에 직접 연결되어 있는지 확인한다.
- 처리: 트랜잭션에서 `WorkbenchSession.projectId`를 `null`로 변경하고, 동일한 프로젝트와 대화를 참조하는 `ProjectItem`을 삭제한다.
- 보존: 원본 `WorkbenchSession`과 결과, 첨부파일 등 대화 종속 데이터는 삭제하지 않는다.

영구 삭제는 기존 `DELETE /api/sessions/[id]`를 재사용한다. 이 API는 사용자 소유권을 검사한 뒤 원본 세션을 삭제하며 Prisma 관계의 기존 연쇄 삭제 규칙을 따른다.

## 클라이언트 상태

`ProjectDetailClient`에 선택된 대화, 모달 단계, 처리 중인 동작, 오류 상태를 둔다. 연결 해제나 영구 삭제가 성공하면 서버 재조회 없이 해당 대화를 `project.sessions`에서 제거하고 `_count.sessions`를 1 감소시킨다. 실패하면 목록을 유지하고 화면 상단 오류 영역에 실패 메시지를 표시한다.

## 오류와 안전성

- 존재하지 않거나 다른 사용자의 프로젝트/대화에는 `404`를 반환해 리소스 존재 여부를 노출하지 않는다.
- 이미 연결 해제된 대화도 `404`로 처리한다.
- 영구 삭제는 위험 버튼과 별도 재확인 단계로 연결 해제와 시각적으로 구분한다.
- 요청 실패 시 모달을 닫지 않고 재시도하거나 취소할 수 있게 한다.

## 테스트

- 연결 해제 요청이 올바른 프로젝트·사용자·대화 조건과 트랜잭션 변경을 구성하는지 테스트한다.
- 다른 사용자 리소스와 다른 프로젝트에 연결된 대화를 거부하는 경로를 테스트한다.
- 클라이언트 선택 상태가 연결 해제, 영구 삭제, 취소를 서로 구분하고 성공 시 목록 및 개수를 갱신하는지 테스트한다.
- 타입 검사, ESLint, 프로덕션 빌드를 실행한다.
- 실제 브라우저에서 모달 선택, 연결 해제 후 프로젝트 목록 반영, 영구 삭제 재확인, 모바일 버튼 배치를 확인한다.

## 범위 제외

- 휴지통 또는 삭제 복구 기능
- 여러 대화 일괄 삭제
- 기존 전체 세션 목록 화면의 삭제 흐름 변경
