## 작업 요약
1~3단계 순차검토체인에서 2~3단계를 반복할 때, 첫 반복 시작 단계(4단계)에서 실행이 멈추고 카드만 남는 문제를 구조적으로 분석하고 수정했습니다. 핵심 원인은 반복 확장 계산이 아니라, `WorkflowStep`와 `Result`가 서로 다른 시점에 생성되면서 실행 인스턴스가 반쯤만 저장될 수 있었던 설계였습니다.

## 변경 사항
### 추가한 기능
- `Result` 생성 시 `WorkflowStep`를 nested create로 함께 만들 수 있는 원자적 실행 생성 경로를 추가했습니다.
- 스트림 이벤트 전송 실패가 durable workflow 본체를 중단시키지 않도록 안전 전송 경로를 추가했습니다.

### 수정한 사항
- 병렬 실행, 병렬 증분 실행, 순차 실행, 순차 증분 실행 모두에서 `WorkflowStep`를 미리 따로 생성하지 않고 `Result` 생성과 함께 묶이도록 변경했습니다.
- 순차 증분 실행의 품질 기준 체크가 템플릿 step의 `orderIndex`가 아니라 실제 확장 실행 순번(`executionOrder`)을 기준으로 동작하도록 수정했습니다.
- 실행 진행 스트림 전송 및 실행 진행도 업데이트 실패가 전체 실행 실패로 전파되지 않도록 보강했습니다.

### 제거/정리한 사항
- 실행 전 단계에서 독립적으로 생성되던 orphan `WorkflowStep` 생성 흐름을 제거했습니다.

## 근본 원인 분석
### 직접 원인
- 반복 실행의 4단계는 DB에 `WorkflowStep(orderIndex=4)`만 생성되고, 같은 실행 순번의 `Result(executionRunId, executionOrder)`와 `AiRequest`는 생성되지 않았습니다.
- 따라서 UI는 “실행 중” 카드가 남지만, 실제 모델 호출은 시작되지 않은 유령 실행 상태가 발생했습니다.

### 구조적 원인
- 기존 설계는 하나의 실행 인스턴스를 `WorkflowStep`와 `Result` 두 테이블에 나눠 기록했습니다.
- 이 두 기록이 원자적으로 묶여 있지 않아 `WorkflowStep` 생성 이후 예외, 재시도, 스트림 실패, 중단 요청, 워커 재개 타이밍 차이가 생기면 반쯤 저장된 실행이 남을 수 있었습니다.
- 반복 구간은 실행 단계 수가 늘어나고 재개/중단/감시 이벤트가 더 많이 개입하므로 이 구조적 약점이 4단계부터 더 자주 드러났습니다.

### 재발 방지 설계
- 실행 인스턴스의 정본은 `Result(executionRunId, executionOrder)`로 보고, `WorkflowStep`는 그 `Result`와 동시에 생성되도록 변경했습니다.
- 즉, 앞으로는 `Result` 없이 `WorkflowStep`만 먼저 남는 상태가 새로 생기지 않습니다.
- 스트림 출력은 관찰 계층으로만 취급하고, 스트림 실패가 실행 본체를 중단시키지 않게 분리했습니다.

## 남은 이슈 / 후속 작업
- 기존에 이미 생성된 orphan `WorkflowStep` 레코드는 과거 세션에 남아 있을 수 있으므로, 별도 정리 스크립트나 마이그레이션으로 후처리하면 더 깔끔합니다.
- 실행 인스턴스 전용 테이블(`ExecutionStep` 또는 유사 모델)로 템플릿 step과 실행 step을 완전히 분리하면 구조가 더 단단해집니다.
